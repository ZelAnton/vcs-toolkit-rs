> rejections. The first group migrated here from `ROADMAP.md`'s former "Deliberately out
> of scope" and "Consciously rejected" lists (so the roadmap holds only live work); the
> last group are new verdicts from the 2026-06-09 development sweep. Recorded so they

## Out of scope (boundaries — belong to a different layer)

| # | Candidate | Why not |
|---|---|---|
| W1 | Copy-on-write worktree cloning (reflink) + its cross-process lock | Stays in `agent-workspace`: the copy strategy is injected by the consumer, and `reflink-copy` is not a toolkit dependency. The toolkit's seam is `worktree_add(…, no_checkout)`, which already exists. |
| W2 | A single cross-backend `merge`/`undo` button on the facade | git merge vs jj `new_merge`+`squash`, and git history rewriting vs jj's op log, diverge for real; §2 exposes honest per-backend primitives instead. |
| W3 | A blocking (non-async) API | Both consumers run tokio; the only synchronous need is `Drop`-context cleanup, which the `blocking` helper modules already cover. |
| W4 | Index-repair / batching policy after `--no-checkout` | Application policy (progress UI, thresholds), not a CLI-wrapping primitive. |

## Resolved by analysis (2026-06-10 roadmap wave)

- **R8 — adopt `Command::ok_codes` — not worth it.** processkit 0.9.1 added `ok_codes`
  (treat a known non-zero exit as *success* for the checking verbs `run`/`run_unit`).
  vcs-toolkit has **no site that needs it**: 0/1 answers use `.probe()`
  (`diff_is_empty`, `branch_exists`, the range/staged checks), and the auth bool uses
  `.exit_code()` (github/gitlab) or `.output_string()` + JSON (gitea) — all already idiomatic
  and clearer than `run()` + `ok_codes(...)`. Converting them would be a step backwards.
  Recorded so it isn't re-attempted; reopen only if a real site wants a non-zero exit to
  count as success for `run()`/`run_unit()`. (R9 `is_transient` and R10 `timeout_grace`
  from the same 0.9.1 surface *were* adopted — see `ROADMAP.md`.)

## New verdicts (2026-06-29)

| # | Candidate | Why not |
|---|---|---|
| W17 | A **Jenkins** backend (and any external CI/CD server — TeamCity / CircleCI / Drone / Buildkite) | Neither a forge nor a VCS, so it fits neither model the toolkit unifies. It has none of the repo/PR/issue/release shape `ForgeApi` codifies — a backend would return `Unsupported` for ~all 17 methods (a degenerate, dishonest backend, worse than Gitea's 4) — and it has **no first-class CLI** in the `gh`/`glab`/`tea` mold (only `jenkins-cli.jar` over SSH/HTTP, or the REST API), so it sidesteps the toolkit's entire pattern (`managed_client!` + the `ProcessRunner` CLI seam). CI is already modeled as **forge-integrated** checks (`pr_checks → CiStatus`/`CheckRun`), and Jenkins doesn't map to it: it's keyed by job name + build number, not by PR. Where Jenkins reports back to GitHub/GitLab as commit statuses/checks (the common setup), `pr_checks` **already surfaces those results** — so native support is largely redundant for the "CI status of a PR" use case. **Right shape if ever needed:** a separate, *non-`vcs-*`* crate over the Jenkins REST API with its own jobs/builds/queue model — not a `ForgeApi` backend — with cross-source CI aggregation left to the consumer (e.g. agent-workspace composing `pr_checks` with a Jenkins client). This sits **further** out of scope than the already-deferred new-forge / new-VCS exercises in `ideas/later-new-backends.md` (those at least share a model). Reopen only with a concrete consumer, and even then as a sibling crate, never a `ForgeApi` backend. |

## Resolved by analysis (2026-07 audit)

- **P2 — keep `vcs-core`'s worktree-delete synchronous; no `spawn_blocking`.** The
  2026-07 audit (`docs/audit-2026-07.md` P2) flagged that jj `remove_worktree` /
  `cleanup_worktree_blocking` call `std::fs::remove_dir_all` **directly inside the
  `async fn`**, so a multi-GB worktree delete briefly stalls the caller's task. The
  obvious fix — `tokio::task::spawn_blocking` — was **declined**: it would make
  `vcs-core` depend on tokio at runtime, but the facade is deliberately
  **runtime-agnostic** (tokio is a dev-dependency only; it must run under any executor
  — tokio current-thread, async-std, smol). Coupling the facade to one runtime is a
  worse trade for a widely-embeddable library than a brief stall on a pathologically
  large delete. Left synchronous with an explaining comment at the call site (same
  spirit as W3 — no blocking API on the facade). Reopen only if `vcs-core` takes a
  runtime dependency for another reason.

## Note

The throughline: vcs-toolkit is a **typed, thin CLI wrapper** over the *installed*
git/jj/gh/glab/tea binaries — it reflects each tool's exact behavior, config, and
credentials, and stays honest about where the backends genuinely diverge (the facades are
a deliberate least-common-denominator, not a false unification). Features that erode that
identity — a blocking API, a false cross-backend button, in-process object access (that's
`gitoxide`/`git2`, see `docs/positioning.md`) — stay out, even when tempting.
