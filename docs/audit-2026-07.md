## How to read the severities

- **CRITICAL** — silent data loss, repo corruption, or credential/security breach in a
  *plausible* call.
- **HIGH** — wrong result or hard failure in realistic, non-exotic use (wrong repo, dead
  safety feature, false "not found").
- **MEDIUM** — wrong behavior on a real but narrower edge (specific state, locale, path
  shape, version).
- **LOW** — papercut, doc drift, or diagnostics-quality issue.

A recurring tension: this is a **thin wrapper over the installed CLIs**, so some findings
are "the CLI genuinely behaves this way" — those get a **`[document]`** disposition (make
the contract honest) rather than **`[fix]`** (the wrapper is actively wrong). Both are
tracked; the distinction is called out per item.

## CRITICAL

### C1 — `Repo::remove_worktree` deletes the working tree / un-snapshotted edits on jj `[fix]`
`crates/core/src/jj_backend.rs:412-421` ✔ (found independently by 3 auditors)

The jj path ignores its `force` parameter (`_force`) and unconditionally
`std::fs::remove_dir_all`s the resolved directory *before* `workspace forget`. The git path
maps `force=false` to a refusing `git worktree remove`.

- **Scenario A (repo wipe):** served over MCP with `--allow-write` (or `--allow-tools
  repo_remove_worktree`), an agent calls `repo_remove_worktree({"path":"."})` (or `""`, or
  the absolute repo path). `dir.join(".")` resolves to the repo root, `workspace_name_for_path`
  matches the **default** workspace, and `remove_dir_all(<repo root>)` wipes `.jj`/`.git`/all
  files. Git is immune (it refuses to remove the main tree); jj is not, because the code
  deletes the directory manually first.
- **Scenario B (silent edit loss):** `remove_worktree(path, false)` on a jj workspace holding
  edits made since the last jj command ran there (jj only snapshots on command in that
  workspace) → git would error "contains modified or untracked files"; jj silently deletes
  them.
- **Fix direction:** honor `force` on jj (probe the workspace for un-snapshotted / untracked
  content and refuse when `!force`); refuse to remove the **default/main** workspace outright;
  resolve+validate `path` is a *secondary* workspace root before any `remove_dir_all`. The
  same sink exists in `cleanup_worktree_blocking` (Drop path) and must get the same guard.

### C2 — `Git::checkout` silently reverts unstaged edits via pathspec fallback `[fix]`
`crates/git/src/lib.rs:1095-1099` ✔✔

`checkout()` runs `git checkout <reference>` with **no trailing `--`**. When `reference`
doesn't resolve as a ref but names a tracked path, git falls into *pathspec* mode and
restores those files from the index, discarding unstaged edits (verified: `git checkout
notes.txt` → "Updated 1 path", exit 0; adding `--` turns it into a hard error).

- **Scenario:** a branch was deleted, then agent-agnostic code calls `checkout(dir, "docs")`
  where `docs/` is a tracked directory → every unstaged edit under `docs/` is irrecoverably
  reverted and the call returns `Ok`. Through `switch_with_stash` the same collision makes the
  helper report a successful "switch" while still on the original branch.
- **Fix direction:** insert the `--` end-of-options/refs separator — `["checkout", reference,
  "--"]` — or force ref-only resolution. Audit **every** positional-taking git verb for the
  same missing `--` (see H3/M-cluster below): `checkout_detach` already isolates via `--detach`,
  but `rev_parse`/`upstream` have the sibling class of "argument that names a path" bugs.

### C3 — jj conflict `resolve()` corrupts every choice on a missing-trailing-newline conflict `[fix]`
`crates/jj/src/conflict.rs:125` ✔✔ (reproduced on jj 0.42)

`sides()`/`base()`/`resolve()` ignore jj's missing-trailing-newline encoding — the
`(no terminating newline)` section label and the phantom `" \n"` context line jj emits.

- **Scenario:** base `"line 1\nline 2\n"`, side-a `"line 1\nmain 2"` (no EOL), side-b
  `"line 1\nfeat 2\n"`. `resolve(Side(0))` yields `"line 1\nmain 2\n"` (a newline side-a never
  had); `resolve(Side(1))` and `resolve(Base)` each gain a spurious trailing blank line. Every
  resolution choice writes back byte-inexact content that then shows as a phantom diff / breaks
  a `render(parse(x))==x` consumer.
- **Fix direction:** parse and honor the `(no terminating newline)` marker per section; drop
  the phantom `" \n"` trailing context line when reconstructing a side from its diff. Add a
  proptest fixture covering EOF-without-newline on each side independently (the existing
  round-trip proptest passes because `render` faithfully reproduces the *input*; the bug is in
  the *reconstruction*, which the round-trip test doesn't exercise).

## HIGH

### H1 — jj bookmark/push/fetch methods glob-match caller strings (mass mutation) `[fix]`
`crates/jj/src/lib.rs:833, 907, 914, 796, 824, 1207` ✔✔ (reproduced on jj 0.42)

jj treats `<NAMES>`/`-b <BRANCH>`/`--remote <REMOTE>` arguments as **string patterns
(globs)**, but the guard only rejects leading-`-`/empty. So a caller string of `*` (or any
name containing `*`) fans out:
- `bookmark_delete(dir, "*")` → "Deleted N bookmarks" (**all** local bookmarks; propagates to
  remote on next push).
- `bookmark_move(dir, "m*", "@", …)` → moves every matching bookmark.
- `git_push(dir, Some("*"))` → pushes **all** bookmarks instead of erroring.
- `bookmark_track` / `git_fetch_from` / `git_fetch_branch` → same shape.
- **Scenario:** a UI/bot forwards a user-supplied bookmark name; `"*"` or a round-tripped
  listed name like `foo*` deletes/moves/pushes unrelated bookmarks.
- **Fix direction:** for the literal-name methods, disable glob interpretation. jj exposes
  exact-match via the `exact:` string-pattern prefix (e.g. `delete exact:"<name>"`), or validate
  the name against jj's bookmark-name grammar (reject `*`/`?`/pattern-prefixes) before spawn.
  This is the jj twin of the git `+main`/refspec-syntax hole (M-cluster) — treat as one
  "typed names must not smuggle CLI-pattern syntax" workstream.

### H2 — Lock-contention retry is dead on jj and locale-fragile on git `[fix]`
`crates/cli-support/src/lib.rs:384-389` ✔✔ + `crates/git/src/lib.rs` mutators not `LC_ALL=C`

Two compounding defects in the opt-in `with_retry(RetryPolicy::lock_contention())`:
1. **jj markers match nothing jj emits.** The constants are `"failed to lock the working
   copy"` / `"failed to lock op heads"`; jj actually prints **"Failed to lock working copy"**
   and **"Failed to lock operation heads store"** (verified against jj source). The retry is
   pure dead code — and the hermetic tests pin the invented strings, so nothing catches it.
   (Compounding: modern jj *blocks* on the lock rather than failing, so contention surfaces as
   a hang/timeout, not a classifiable exit at all.)
2. **git markers are English-only, on un-normalized commands.** `LC_ALL=C` is forced only on
   the ~19 *classifier-feeding* commands (commit/merge/fetch/stash-pop/…), **not** on `add`,
   `checkout`, `reset_*`, `branch`/`tag`/`config`/`remote`/`worktree_*`, `push`, `init`. git
   translates "Unable to create '…index.lock': File exists" (de: "Konnte '…' nicht
   erstellen…"), so on a non-English machine the marker never matches for exactly the
   index.lock-prone commands.
- **Scenario:** German/Russian/Chinese CI runner, `Git::with_retry(...)`, concurrent
  `git add`/`checkout` racing on `index.lock` → the configured retry silently never fires; a
  `Jj::with_retry(...)` user gets *no* jj lock retry ever, on any locale.
- **Fix direction:** correct the jj marker strings (and delete/repair the tests that pinned the
  wrong ones); either force `LC_ALL=C` on all mutating commands whose failures the retry
  classifies, or match on the locale-stable fragment only (the literal `index.lock` path
  substring, `.lock': ` is translated — the *path* is not). Re-evaluate whether jj's
  block-on-lock behavior means jj lock-retry should be dropped and documented instead.

### H3 — `harden()` silently provides no hardening on git < 2.31 `[fix]`
`crates/git/src/lib.rs:2045-2062` vs the major-only gate at `:531-537` ✔ (2 auditors)

`harden()` pins hooks-off / fsmonitor-off / sshCommand-off through
`GIT_CONFIG_COUNT`/`KEY_n`/`VALUE_n`, which git only reads since **2.31**. Older git ignores
the vars with no error. The only version gate accepts any 2.x.

- **Scenario:** hardened client driving an untrusted checkout on Ubuntu 20.04 (git 2.25) or
  RHEL 8 (2.27) → `harden()` returns normally but repo-local hooks / `core.fsmonitor` /
  `core.sshCommand` all still execute — arbitrary code execution under a client the caller
  believes is sandboxed. The ≥2.31 requirement exists only in a doc comment.
- **Fix direction:** probe `git --version` in `harden()`/`hardened()` and *fail loudly* (or
  fall back to a `-c key=value` per-command config injection, which works on older git) when
  below the floor. More broadly, raise `MIN_SUPPORTED_MAJOR` to a real `(major, minor)` floor
  (see M "version gate too low").

### H4 — Default client doesn't scrub `GIT_DIR`/`GIT_WORK_TREE`/`GIT_INDEX_FILE` → wrong-repo mutation `[fix]`
`crates/git/src/lib.rs:2012-2038` (scrub lives only inside `harden()`) + `crates/core/src/lib.rs:314` ✔

`Repo::open` builds a plain `Git::new()`, and `command_in` uses `current_dir(dir)` rather
than `git -C <dir>`. The repo-redirecting env vars inherited from the parent win over the
bound directory, and they're only removed by opt-in `harden()`.

- **Scenario:** a consumer tool built on `Repo::open` is invoked from inside a git hook (git
  exports `GIT_DIR`; commit-path hooks export `GIT_INDEX_FILE`) or from a shell with `GIT_DIR`
  exported, while targeting a *different* repo → every toolkit command reads and **mutates**
  the hook's repo/index instead of `dir`: a `reset --hard`/`checkout` lands in the wrong
  repository with no error.
- **Fix direction:** always pass `git -C <dir>` (defeats `GIT_DIR` precedence for the working
  directory) and scrub the repo-redirectors (`GIT_DIR`, `GIT_WORK_TREE`, `GIT_INDEX_FILE`,
  `GIT_COMMON_DIR`, `GIT_OBJECT_DIRECTORY`, `GIT_NAMESPACE`) on the **default** client, not
  only under `harden()`. These are correctness, not just hardening.

### H5 — Inline credential helper leaks the token to any host git contacts `[fix]`
`crates/cli-support/src/credentials.rs:360` (installed via `with_credentials`/`with_token`) ◦

The inline `credential.helper` answers git's `get` action unconditionally, ignoring the
`host=`/`protocol=` request on stdin, so the secret is handed to **every** HTTPS host the
operation touches.

- **Scenario:** `git.with_token(t).fetch(dir)` where the fetch follows a server redirect, or
  recurses on demand into a submodule whose `.gitmodules` URL points at `https://attacker.com`
  → the PAT is transmitted to the attacker's server. A host-scoped ambient helper would not
  release it.
- **Fix direction:** make the helper echo the token only when the requested `host`
  (and `protocol=https`) matches the intended remote's host — read git's key=value request on
  stdin and gate on it. Bind the credential to the operation's actual remote host at
  construction time.

### H6 — `parse_conflicts` rejects files whose *content* contains marker-like lines `[fix]`
`crates/git/src/conflict.rs:117` and `crates/jj/src/conflict.rs:190` ✔✔ (jj reproduced on 0.42)

Both parsers hard-error (`Error::Parse`, "marker outside a region" / "use vcs_git::conflict")
on any out-of-region line that is a bare run of ≥7 `=`/`>`/`<`. But those are ordinary
content: Markdown/RST setext underlines (`=========`), `=======` divider banners, deep
`>>>>>>>` email quotes, or a file that literally documents conflict markers. The docs claim
such lines "are not mistaken for a separator" — the code contradicts them.

- **Scenario:** the documented `conflicted_files → parse_conflicts → resolve` workflow fails
  with `Error::Parse` on a conflicted `README.rst` whose (unconflicted) title is underlined
  with `=========`, or a jj file that contains a `<<<<<<<` documentation line alongside a real
  conflict — programmatic resolution becomes impossible on files git/jj merged fine.
- **Related (M1):** even *inside* a region, the first bare `=`-run of matching length is taken
  as the separator even when it's content, so `resolve()` silently returns corrupted text while
  `render()` stays byte-exact (masking it). `crates/git/src/conflict.rs:156`.
- **Fix direction:** only treat a marker-run as structural when it appears where the grammar
  expects it (a `<<<`-opened region for `=`/`>`; top-level otherwise is content). Track region
  state strictly and never classify an out-of-region run as an error. Add fixtures with
  marker-like content for both backends.

### H7 — `show_file`/`file_show`/`diff_text` strip trailing whitespace via `trim_end()` `[fix]`
`crates/git/src/lib.rs:1796` · `crates/jj/src/lib.rs:1116` · `crates/git/src/lib.rs:1408` ✔✔

These read blob/file content through `core.run`, which `trim_end()`s stdout — silently
deleting the blob's trailing newline(s) and trailing whitespace. Only binary-lossiness is
documented, not this. (`diff_text` additionally drops a final single-space context line,
desyncing the last hunk from its `@@` count.)

- **Scenario:** fetch a file at a rev to restore it (or grab a conflict stage per the conflicts
  guide) and write it back → the file loses its terminating newline and any trailing blank
  lines, producing spurious "\ No newline at end of file" diffs and a byte-inexact round trip.
- **Fix direction:** these content-returning verbs must use the **raw** capture (`run_raw` /
  `output_string` without `trim_end`), not the trimmed `run`. Audit every content-vs-scalar
  verb: scalars (ids, branch names) want trim; file/blob/diff/template bodies must not be
  trimmed (`template_query` at `jj/lib.rs:1035` has the same defect).

### H8 — Gitea list/view silently clamped to 50 rows → false "not found" `[fix/document]`
`crates/gitea/src/lib.rs:439, 473` (+ issue_list `:570`, release_list `:613`) ✔✔ (verified vs Gitea server)

Gitea's server clamps API page size to `MAX_RESPONSE_ITEMS` (default **50**) and `tea` makes
a single call, so `--limit 100`/`--limit 999` never take effect. Two consequences:
- `pr_view(dir, N)` re-lists all PRs and searches; for a PR outside the ~50 most recent it
  returns a confident false `"no pull request #N"` `Error::Parse`, and the truncation-hint at
  `:481` is dead code.
- `pr_list`/`issue_list`/`release_list` top out at 50 with no error and no signal — exactly the
  silent truncation the pinned `--limit` comment claims to prevent.
- **Scenario:** default-configured Gitea/Forgejo (incl. gitea.com) with >50 all-state PRs or
  >50 open issues.
- **Fix direction:** `pr_view` must not rely on a single capped list. (Correction: `tea` has
  **no** single-PR view — `tea pulls`/`pr` exposes only list/checkout/create/close/merge/… , no
  `view`/index subcommand — so a direct fetch is *not* an option, unlike `issue_view`'s
  `tea issues <index>`.) It therefore **pages** `tea pr list --state all --page N` (a documented
  `tea` flag) until the PR is found or a page returns empty. For the list verbs, **document** the
  ~50-row server ceiling honestly and drop the misleading `--limit 100` comment.

### H9 — `gh api` / `glab api` ignore the bound directory → wrong-repo resolution `[fix]`
`crates/github/src/lib.rs:688` and `crates/gitlab/src/lib.rs:422` ✔

Both run the escape-hatch `api()` via plain `run` (not `command_in(dir, …)`) and forward it in
the `bare` block of the `*At` view, yet gh/glab resolve `{owner}/{repo}`/`:id`/`:namespace`
endpoint placeholders from the **process cwd's** git remotes.

- **Scenario:** a multi-repo orchestrator (the crate's dir-taking design target) whose process
  cwd is repo A calls `gh.at(repo_b).api("repos/{owner}/{repo}/pulls")` → the placeholders
  resolve to repo A; the call silently reads/mutates the wrong repository (or errors "not a git
  repository" when cwd is none).
- **Fix direction:** bind `api()` to `dir` like every other verb (`command_in(dir, …)`), and
  remove it from the `bare` forwarder group so `*At` runs it in the bound dir.

### H10 — `Forge::supports()` claims every op works on an `Unknown` handle `[fix]`
`crates/forge/src/lib.rs:307-316` ✔

`supports()` special-cases only Gitea; the catch-all `_ => true` returns `true` for **all**
`ForgeOp`s on a `ForgeKind::Unknown` handle — while every operation on `Unknown` returns
`Error::Unsupported`.

- **Scenario:** an auto-detector builds the documented `from_unknown(cwd)` "tried, no luck"
  path; a UI renders availability from `ForgeOp::ALL` + `supports()` → all buttons show
  available, each click fails `Unsupported`. The invariant test `supports_matches_unsupported_ops`
  pins exact agreement but never covers `Unknown`.
- **Fix direction:** `supports()` must return `false` for all ops when `kind() ==
  ForgeKind::Unknown` (and extend the invariant test to include `Unknown`).

### H11 — jj `Repo::checkout` amends/rewrites the checked-out commit `[document]`
`crates/core/src/jj_backend.rs:234-242` ◦

The facade maps `checkout("ref")` → `jj edit <ref>`, which makes the *bookmarked commit
itself* the working-copy change. Subsequent facade ops rewrite it rather than build on top.

- **Scenario:** backend-agnostic code does `checkout("main")` → edit a file →
  `commit_paths(["f"], "fix")`; on git this appends a commit to `main`, on jj it amends the
  existing `main` tip's tree *and* replaces its description — a silent rewrite of a possibly-
  pushed commit. Undocumented on `Repo::checkout`.
- **Fix direction:** this is a genuine backend divergence, so **document** it prominently on
  `Repo::checkout` (and the cookbook), and consider offering a `checkout_new`/"start work on
  top of ref" primitive (`jj new <ref>`) as the git-`checkout`-like alternative, mirroring how
  `transaction` documents its non-closure path.

## MEDIUM

Grouped by theme; each is a real edge with a concrete trigger.

### jj path/name model
- **M2 — `JjFileset::path` is documented root-relative but jj `file:` is cwd-relative.**
  `crates/jj/src/lib.rs:214` ✔✔. Every fileset consumer (`file_show`, `commit_paths`,
  `squash_paths`, `split_paths`, `absorb`) targets the wrong file when `dir` ≠ workspace root;
  worse, a same-named file under `dir` gets silently operated on. **Fix:** resolve filesets
  relative to the workspace root (prefix with `root:` or compute the relative path from root),
  or document `dir` must be the workspace root.
- **M3 — `status`/`diff_summary` return cwd-relative paths (incl. `..`).**
  `crates/jj/src/lib.rs:691` ✔✔. From `<repo>/sub`, output is `M ..\root.txt` — no
  root-relative comparison (or the `--git` DTO the docs claim parity with) ever matches. **Fix:**
  run diff from the workspace root or post-process to root-relative.
- **M4 — `JjFileset::path` unconditionally rewrites `\`→`/`.** `crates/jj/src/lib.rs:215` ✔.
  Corrupts legitimate Unix filenames containing backslashes (git twin does this
  `#[cfg(windows)]`-only, with a comment). **Fix:** gate the rewrite on `#[cfg(windows)]`.
- **M5 — bookmark names with commas/spaces mis-split.** `crates/jj/src/lib.rs:306` &
  `crates/jj/src/parse.rs:283` ✔✔. jj renders exotic names quoted; splitting on `,`/whitespace
  yields phantom names (`"a` etc.) that then break `git_push -b`. **Fix:** use a NUL/newline
  record separator in the template, or parse the quoted form.

### jj semantics
- **M6 — `rebase` defaults to `-b @` (whole branch).** `crates/jj/src/lib.rs:1127` ✔✔. Doc says
  "rebase the working copy"; `jj rebase -d X` moves fork-point siblings of `@` too. **Fix:** pass
  `-r @` (or `-s @`) for the documented single-change semantics, or document the `-b` behavior.
- **M7 — `commit_paths` with empty filesets commits the entire working copy.**
  `crates/jj/src/lib.rs:1157` ✔. Doc promises "exactly these filesets"; empty slice → bare
  `jj commit -m`. **Fix:** refuse an empty fileset slice (like `split_paths` already does).
- **M8 — `try_merge`/`transaction` op-restore reverts concurrent actors.**
  `crates/core/src/jj_backend.rs:256-277`, `crates/jj/src/lib.rs:1485` ✔. `op restore <pre>`
  restores the whole repo view, silently reverting a concurrent `jj describe`/`bookmark move`
  landed between capture and restore. Compounding: if the closure's `Err` was a fired
  `default_cancel_on`, the restore is itself cancelled → no rollback. **Fix:** document the
  concurrent-actor caveat; for cancellation, run the restore with a fresh (uncancelled) token.
- **M9 — `workspace_roots` errors for pre-0.38 workspaces → leak.** `crates/jj/src/lib.rs:1410`
  ◦. `jj workspace root --name` "errors out" for workspaces created before 0.38, so
  `remove_worktree`/Drop-cleanup reports not-found and leaks them. **Fix:** fall back to
  `workspace list` parsing when `--name` errors.
- **M10 — blocking `workspace_name_for_path` snapshots (mutation) and returns `None` on lock.**
  `crates/jj/src/lib.rs:1629` ◦. Runs `workspace list`/`root` without `--ignore-working-copy`
  from a Drop guard → mutates the copy N+1 times and silently leaks on lock contention. **Fix:**
  add `--ignore-working-copy`; surface the error instead of collapsing to `None`.

### git output/state
- **M11 — porcelain ` R`/` C` worktree-renames misparsed.** `crates/git/src/parse.rs:120` ✔✔.
  `parse_porcelain` only consumes the rename-source record when the *first* byte is `R`/`C`;
  git also emits ` R`/` C` (worktree rename), whose source record becomes a phantom entry with
  garbage `path`/`code`. **Fix:** check either XY column for `R`/`C`.
- **M12 — `switch_with_stash` targets the wrong stash / flattens the index.**
  `crates/git/src/lib.rs:2088-2098` ✔✔. `stash push -u` exits 0 saving nothing (e.g. submodule-
  only dirt), then bare `stash pop` splats an unrelated pre-existing (or another actor's)
  stash; and `pop` without `--index` returns everything unstaged. **Fix:** capture the pushed
  stash ref (`stash create`/`stash store`, or check `stash push` actually saved) and pop that
  specific entry with `--index`; document the single-actor contract. **Landed (2026-07):** the
  stash-list depth is checked around the `push` so a no-op push doesn't pop an unrelated stash,
  the pop uses `--index`, and the single-actor contract is documented.
- **M13 — `rev_parse` without `--verify` returns a filename as a fake commit id.**
  `crates/git/src/lib.rs:1052` ✔✔. `git rev-parse Makefile` → `Makefile`, exit 0. **Fix:** add
  `--verify` (as `rev_parse_short`/`resolve_commit` already do).
- **M14 — `harden()` env-scrub misses documented `GIT_*` vectors.** `crates/git/src/lib.rs:2013`
  ◦. Omits `GIT_PROXY_COMMAND` (arbitrary program for `git://`), `GIT_COMMON_DIR`,
  `GIT_EXEC_PATH`, `GIT_TEMPLATE_DIR` (hook-installing templates), `GIT_*_PATHSPECS`. **Fix:**
  extend the scrub list (pairs with H4).
- **M15 — `GitAt.run_args`/`run_raw_args` execute in the process cwd, not the bound dir.**
  `crates/git/src/lib.rs:2151` ✔. The `bare` forwarder group violates `GitAt`'s "drops the
  leading dir" contract for the escape hatches. **Fix:** move the `run*_args` forwarders to
  inject `-C dir` / `command_in`, or document that escape hatches are cwd-bound.
- **M16 — push refspec syntax rides through the name guard (`+main` force-push).**
  `crates/git/src/lib.rs:1508` ✔. A leading `+` makes the push a force-push; `RefName::new`

  (jj twin: H1).

### facade correctness
- **M17 — gone upstream reported as `↑0↓0` (in sync).** `crates/core/src/git_backend.rs:138-144`

- **M18 — `snapshot().dirty` and `has_uncommitted_changes()` disagree on conflicted-empty `@`.**
  `crates/core/src/jj_backend.rs:71-76` ✔. The 0.4.0 "conflicted ⇒ dirty" fix was applied only
  to `snapshot()`. **Fix:** apply the same `|| conflicted` to the boolean methods.
- **M19 — `detect` lets any `.jj` dir beat a valid `.git`.** `crates/core/src/lib.rs:230` ✔. Only
  `is_dir()` is checked (comment claims symmetry with the validated `.git` probe). A stray/empty

- **M20 — `in_progress_state` misses cherry-pick/revert/bisect; conflates `git am` with rebase.**

### environment / scale
- **M29 — git version gate is major-only while argv needs ≥2.11/2.13/2.31.**
  `crates/git/src/lib.rs:531-537` ✔. git 2.7 passes `ensure_supported()`, then

  the crate's own argv requires).
- **M30 — unbounded output capture → OOM.** processkit `OutputBufferPolicy::unbounded()` default,

  `with_max_bytes`/`fail_loud`).
- **M31 — non-UTF-8 filenames/messages lossily replaced (U+FFFD) → wrong path.** processkit

### watch
- **M32 — watch re-query is not read-only on jj.** `crates/watch/src/lib.rs:502` ✔. `snapshot()`/

- **M34 — a re-query skipped on timeout is never retried → change lost forever.**

## Synthesis — the recurring themes

Most findings collapse into a handful of systemic patterns; fixing the pattern fixes the
cluster:

1. **Caller strings smuggle CLI meta-syntax past the guard.** `reject_flag_like` stops only
   leading-`-`/empty, but jj glob-patterns (H1), git refspec `+`/`:` (M16), gitea/jj bare
   positionals (L2), and glab's `-`-means-editor (M24) all pass through. → One "typed name/ref
   validation" workstream: per-backend, reject the metacharacters that backend interprets.
2. **`trim_end()` on content verbs.** `show_file`/`file_show`/`diff_text`/`template_query`
   corrupt trailing bytes (H7). → Split "scalar" (trim OK) from "content/body" (must be raw)
   capture, once, across both facades.
3. **English-only, un-normalized classification.** Lock/transient/auth markers are substring
   matches on localized CLI prose, and `LC_ALL=C` isn't forced on all classified commands (H2,
   M28); the jj lock markers are simply wrong (H2). → Force `LC_ALL=C` on every command whose
   failure feeds a classifier, match locale-stable fragments, and correct/verify the jj strings.
4. **Directory binding gaps.** `gh/glab api` (H9), `GitAt.run*_args` (M15), and inherited
   `GIT_DIR` on the default client (H4) all let the *process cwd* or a poisoned env win over the
   bound `dir`. → Always `-C dir` / `command_in`, scrub redirectors by default.
5. **Version-floor honesty.** git gate is major-only (M29), forges have none (M25), `harden`
   silently degrades below 2.31 (H3). → Real `(major, minor)` floors with a loud probe.
6. **jj-vs-git contract divergences surfaced as bugs.** `remove_worktree` force (C1), `checkout`
   = `jj edit` (H11), cwd-relative filesets (M2/M3), op-restore blast radius (M8),
   dirty-conflicted disagreement (M18). → Some are `[fix]`, some are `[document]`; the facade's
   "honest LCD" value means each must be *decided*, not left implicit.
7. **Unbounded/lossy IO.** Capture is unbounded (M30) and lossy on non-UTF-8 (M31); acute in the
   long-lived MCP/watch processes. → Wire processkit's `with_max_bytes`/`output_bytes` on the
   at-risk verbs.

## Recommended remediation order

Each wave is independently shippable, gated by the full CI matrix + the project's ≥2-pass
adversarial review, and (per the crates' pre-1.0 status) free to break API where needed.

- **Wave 0 — stop the data-loss / security bleeders (C1, C2, C3, H1, H5).** These silently
  destroy user data or leak credentials on plausible calls; nothing else matters more. Each is a
  small, local change with a clear fix and a missing test to add.
- **Wave 1 — dead/degraded safety features (H2, H3, H4, H6, H10).** Features the docs promise
  that silently don't work (lock retry, hardening, conflict parsing on real files, forge
  capability truth). High "surprise" cost for early adopters.
- **Wave 2 — wrong-result correctness (H7, H8, H9, M11–M13, M16–M21, M29, M30).** Byte-inexact
  reads, wrong-repo/wrong-stash, silent truncation, mis-detected state.
- **Wave 3 — cross-backend contract & CLI-reality (H11, M2–M10, M22–M28, M31–M34).** The
  `[fix]` vs `[document]` decisions: make the facade's least-common-denominator honest, add the
  missing version floors, correct the CLI-behavior contracts.
- **Wave 4 — papercuts & doc drift (all LOW).** Fold into the next release-prep pass; the doc
  fixes (L18) are the same cross-facade class as the previously-fixed `pr_comment` inconsistency
  and should be caught by a docs-vs-code sweep.

> **Publishing gate:** several CRITICAL/HIGH items (C1 via MCP, C2, H5, H2) are the kind of
> silent-data-loss / silent-security regressions that would earn a bad first impression on a
> repo aiming for popularity. Wave 0 + Wave 1 should land **before** the next coordinated
> crates.io release, not after.

## Headline

The runtime code is **unusually panic-disciplined** — **zero** `.unwrap()`/`.expect()`/`panic!`
in non-test library code, all CLI-number parses land in `unwrap_or`/`.ok()`, all byte-slicing
is char-boundary-safe (620 raw grep hits → 19 real sites → all provably-safe or test-only). So
this pass surfaced **no new CRITICAL panics**. The genuinely new severe material is in **API
shape** (bare bools and the input-rejection error model would force breaking changes right
after 1.0) and **packaging** (a `patch` release can ship a broken dependency floor or silent
API removals). Detail below.

## Panic / reliability (dimension P)

### P1 — `Builder::max_wait(Duration::MAX)` panics the watch loop → silent permanent death `[fix]`
`crates/watch/src/lib.rs:472` ✔

`let deadline = tokio::time::Instant::now() + config.max_wait;` uses the panicking
`Add<Duration>` on an **unclamped, caller-settable** ceiling (`Builder::max_wait`, `:190`, no
clamp, docs don't forbid huge values).

- **Scenario:** a caller writes `.max_wait(Duration::MAX)` as the natural "disable the re-query
  ceiling" idiom → the first FS-event burst hits `:472` → "overflow when adding duration to
  instant" panic inside the `tokio::spawn`ed `watch_loop` → the `JoinHandle` swallows it,
  `out_tx` drops, `recv()` returns `None` — the watcher dies **permanently and silently**,
  indistinguishable from a clean teardown. (`sleep(debounce)` at `:491` and `timeout` at `:514`
  are `checked_add`-capped and safe; only this site is raw.)
- **Fix direction:** `checked_add` with a saturating fallback, or clamp/validate `max_wait` in
  the builder.

### P2 — blocking `remove_dir_all` runs on the async worker (jj worktree removal) `[wont-fix]`
`crates/core/src/jj_backend.rs:421` (rollback `:400`) ◦ — pairs with **C1**

> **Decision (Wave 0): not fixed.** The obvious fix — `tokio::task::spawn_blocking` — would
> make `vcs-core` depend on tokio at runtime, but the facade is deliberately
> **runtime-agnostic** (tokio is a dev-dependency only; it must run under any executor —
> async-std, smol, a current-thread runtime). Coupling the facade to one runtime is a worse
> trade for a library meant to be widely embedded than a brief stall on a pathologically large
> worktree delete. Left as-is with an explaining comment at the call site. Revisit only if
> `vcs-core` takes a runtime dependency for other reasons.

The jj `remove_worktree` (and the `create_worktree` failure rollback) call
`std::fs::remove_dir_all` **directly inside the `async fn`**, not via `spawn_blocking`. The git
path shells out asynchronously, so only jj is affected.

- **Scenario:** MCP `repo_remove_worktree` on a jj workspace whose dir holds a multi-GB
  `target/` → the tokio worker is pinned for the whole multi-second recursive delete; a consumer
  on a current-thread runtime freezes **every** task (in-flight requests, watcher debounce
  timers) for the duration.
- **Fix direction:** wrap the fs delete in `spawn_blocking` (fold into the C1 fix, which reworks
  this path anyway).

- **P3** `crates/core/src/jj_backend.rs:143` — `debug_assert_eq!(fields.len(), 3, …)` on jj

## Build / packaging / dependencies (dimension B)

### B1 — intra-workspace version floors don't cover the HEAD API (masked by `path` deps) `[fix]`
`crates/{github,gitlab,gitea}/Cargo.toml` (`vcs-cli-support` floor) + `crates/{git,jj}` (`vcs-diff`) + `crates/core` (`vcs-git`) ✔ (github/Cargo.toml:20-21 confirmed)

Six manifests carry intra-workspace floors that **exclude the API/feature the HEAD code
requires**, masked by the `path` dep so it builds locally:
- github/gitlab/gitea require `vcs-cli-support = "0.2", features=["serde"]`, but published
  **0.2.0 has no `serde` feature** (the `json` module was added after). The manifest even
  comments "its NEXT release must be ≥ minor and this floor raised to match."
- git/jj re-export `vcs_diff::DiffSpec`, absent from published vcs-diff 0.2.0.
- core calls `git.fetch_branch(...)`, which was `fetch_remote_branch` in published vcs-git 0.6.0.
- **Scenario:** an `all` release with the **default `patch` bump** publishes e.g. vcs-github

  floor on a patch bump; the dry-run degrades to plain `cargo build`; the real `publish --verify`
  resolves the just-published 0.2.1 (which *has* the API) so it passes **silently**. A consumer
  whose lock pins cli-support 0.2.0 (or who uses `-Z minimal-versions`/`--precise`) then hits
  "feature `serde` not found" or a downstream compile error inside a crates.io crate.
- **Fix direction:** force these crates' next release to ≥ minor and set the floors explicitly
  (`vcs-cli-support = "0.3"`, `vcs-diff = "0.3"`, `vcs-git = "0.7"`); add a release-workflow
  check that fails when a floor is below the sibling's about-to-publish version.

### B2 — a `patch` release can ship public-API removals; semver-checks isn't a gate `[fix]`
`.github/workflows/release.yml:55-63` (`bump` default `patch`) + `ci.yml` (semver-checks `continue-on-error`) ◦

HEAD contains public-API **removals/renames** since the tags (`fetch_remote_branch → fetch_branch`

`cargo-semver-checks`, is report-only (documented pre-1.0) and runs in CI, not as a release gate.

- **Scenario:** a `patch` release ships vcs-core 0.4.1 / vcs-forge 0.2.1 with removed `pub`
  items inside a semver-compatible range; every existing consumer breaks on their next
  `cargo update` with no opt-out short of pinning `=0.4.0`.
- **Fix direction:** make the release workflow **require** the bump be ≥ minor when
  semver-checks reports a breaking change (promote it from report-only to a release-time gate),
  or compute the minimum legal bump automatically.

### B3 — `cargo package` validates only 3 of 12 crates before the irreversible publish loop `[fix]`
`.github/workflows/ci.yml:250-270` + `release.yml:335-368` ◦

CI `cargo package` covers only vcs-diff/vcs-cli-support/vcs-testkit; in an `all` release the

- **Fix direction:** run `cargo package` (not `build`) for all 12 in dependency order in the
  dry-run, with `--no-verify` off; the job comment's "not yet on crates.io" premise is also stale
  (all 12 are published) and should be corrected.

## Public API / 1.0-readiness (dimension A)

> These are the costliest class: the wrapper/facade **traits freeze at 1.0**, so a bare bool or
> a transposable pair becomes a permanent foot-gun, and reshaping it later breaks every caller,
> every external trait impl, and every `mock`-feature user at once.

### A1 — bare `bool` params baked into the 1.0 facade & wrapper traits `[fix]`
`crates/core/src/lib.rs:505` (`delete_branch(name, force)`), `:741` (`remove_worktree(path, force)`), `crates/forge/src/lib.rs:490` (`pr_close(number, delete_branch)`), + wrapper-trait siblings: `git/src/lib.rs:687,765,776`, `github:458`, `jj:483,555`, `gitea:320` ✔ (remove_worktree/force confirmed via C1) — generalizes **C1**

Violates the workspace's own AGENTS.md rule ("bare positional bools don't pass review"), and each
bool is already **cross-backend dishonest**: jj *ignores* `force` (that's C1); GitLab/Gitea
*ignore* `delete_branch`; `jj git_clone(url, dest, colocate: bool)` is a bare bool where the exact
git sibling got a `CloneSpec` builder.

- **Scenario:** a consumer writes `repo.remove_worktree(p, false)` expecting a safety refusal and
  gets backend-dependent semantics; `stash_push(dir, true)` is uninterpretable at review. When the
  C1 fix gives `force` a real contract, every call site **and** every external `VcsRepo`/`ForgeApi`
  implementer breaks.
- **Fix direction:** replace each with a spec struct (`WorktreeRemove::new(path).force()`,
  `BranchDelete`, `PrClose::delete_branch()`, `JjCloneSpec`) **before** the traits freeze. This is
  the single most schedule-sensitive item: it must land pre-1.0 or never cheaply.

### A2 — input rejection has no stable, matchable home in the error model `[fix]`
`crates/cli-support/src/lib.rs:281` + `crates/core/src/lib.rs:566` + `crates/forge/src/error.rs:27` ◦ — generalizes **L9**

Three encodings coexist: the wrappers fabricate a **fake** `Error::Spawn{source: InvalidInput}`
(from `reject_flag_like`/`RefName::new`/…), vcs-core uses `Error::Io(InvalidInput)`, only vcs-forge
has a real `Error::InvalidInput` — and **no crate ships an `is_invalid_input()` classifier**.

- **Scenario:** a binding (K5: "branch on `is_*`, never string-match stderr") maps `Spawn` to an
  OS/environment exception, so a caller-supplied `"-evil"` branch name surfaces as "failed to
  spawn git" instead of a `ValueError`. Fixing it properly post-1.0 means a per-wrapper error enum,
  re-typing every `Result<T>` in five crates — the most expensive reshape on the surface.
- **Fix direction:** decide the input-rejection contract now — a shared `InvalidInput` variant (or
  a distinct error) reachable via an `is_invalid_input()` classifier on every facade — and route
  `reject_flag_like`/newtype validation through it.

### A3 — "not found" classifier means *binary* missing, never *resource* missing `[fix]`
`crates/forge/src/error.rs:121`, `crates/core/src/error.rs:72` ◦

`is_not_found()` on both facades means "the CLI binary is missing"; there's **no** classifier for
"the PR/issue/release/worktree doesn't exist". core even has an `Error::WorktreeNotFound` variant
its own `is_not_found()` returns `false` for, and a gitea test explicitly pins "no pull requests
found" as **not** not-found.

- **Scenario:** a Python user catches `NotFoundError` around `pr_view(9999)` and never gets it — a
  missing PR is an unclassified `Exit`; they must string-match stderr, exactly what K5 promised they
  wouldn't. Renaming/adding resource-not-found semantics post-1.0 changes observed exception mapping.
- **Fix direction:** add an `is_resource_not_found()` (distinct from the binary-missing sense) and
  make `WorktreeNotFound`/"no such PR" answer it.

### A4 — output DTOs are `#[non_exhaustive]` with no constructor & no `Deserialize`, yet the traits invite external impls `[decide]`
`crates/core/src/dto.rs:111,57,38,94` + `crates/forge/src/dto.rs:154,215,253,195` ◦

`RepoSnapshot`/`WorktreeInfo`/`FileChange`/`UpstreamTracking`/`ForgePr`/… can't be built outside
their crate, but `ForgeApi`'s defaulted methods exist "so external trait implementers keep
compiling" and `RepoWatcher::builder(impl VcsRepo)` invites custom adapters. vcs-watch itself hit
this wall and mirrors the fields into a private `WatchState` (`event.rs:96` — "constructible
in-crate — `RepoSnapshot` is `#[non_exhaustive]`").

- **Scenario:** a consumer writing a `VcsRepo` test double (no `mock` feature on the facades) cannot
  return a `RepoSnapshot` at all.
- **Fix direction:** either add public constructors/builders for the return DTOs, or drop the
  "externally implementable" claim for those traits. Constructors are additive; the incoherence
  should still be *decided* at 1.0.

### A5 — constructor & method signatures with adjacent same-typed params (transposition foot-guns) `[fix]`
`crates/core/src/lib.rs:328` (`from_git(root, cwd, client)` — the K9 FFI seam, frozen forever), `:724` (`create_worktree(path, branch, base)`), `crates/forge/src/lib.rs:534` (`issue_create(title, body)`), `crates/git/src/lib.rs:682` (`is_merged(dir, branch, target)` — swap *inverts* the answer), `:685`, `:800`, jj `:437,494,513` ◦

The `from_git(root, cwd)` transposition already produced a fixed bug; the shape is systemic. Two
adjacent `impl Into<PathBuf>`/`&str` of identical type compile when swapped and silently misbehave
(`create_worktree(p, "main", "feature")` creates a branch *named* `main`; `is_merged` returns a
frequently-`true`-in-tests wrong answer).

- **Fix direction:** spec/newtype the highest-risk ones (`WorktreeCreate::new(path).branch().base()`,
  `IssueCreate` mirroring `PrCreate`, a `Located`-taking constructor overload for the FFI seam)
  before the traits freeze; `issue_create` also can't grow labels/assignees without a spec.

### A6 — "parallel wrappers" diverge in merge-spec shape and DTO honesty `[fix/document]`
`crates/gitlab/src/lib.rs:319` & `gitea:320` (bare `MergeStrategy` vs github's `PrMerge` spec) + `crates/forge/src/dto.rs:168` (prose-only DTO honesty) ◦ — generalizes pass-1 DTO findings

`GitHubApi::pr_merge` takes a `PrMerge` spec (`.auto()`/`.delete_branch()`) while gitlab/gitea take
a bare `MergeStrategy` enum — so a vcs-gitlab consumer can't express `--remove-source-branch`/auto.
Separately, DTO honesty is **prose-only**: `ForgePr.draft: bool` means "unknown" on GitHub/Gitea;

unreported — with no machine-readable per-field "unsupported here" signal (`ForgeCapabilities`
covers whole operations only).

- **Fix direction:** give gitlab/gitea a merge spec matching github's; for DTO honesty, either
  `Option`-wrap the ambiguous fields (breaking — do it pre-1.0) or add a per-field support query.

## Resource / process / concurrency (dimension R)

### R1 — MCP mutating tools aren't serialized against concurrent tool calls `[fix]`
`crates/mcp/src/lib.rs:516` ◦ — generalizes the op-restore blast-radius of **M8**

rmcp 1.7 spawns a task **per request** (`spawn_service_task`), and `VcsMcpServer` (Arc'd handles,
all-`&self` tools) applies no per-repo mutex around multi-step sequences like `repo_try_merge`
(jj: `op_head` → `new_merge` → `op_restore`; git: `merge --no-commit` → `merge --abort`).

- **Scenario:** server `--allow-write` on a jj repo; a pipelining MCP client issues `repo_try_merge`
  and `repo_commit` concurrently → the commit's op lands between try_merge's `op_head` capture and
  its **unconditional** `op_restore` → the restore rewinds past the concurrent commit, silently
  discarding it while both tools report success.
- **Fix direction:** a per-repo async `Mutex` around the mutating tools (or at least the compound
  ones), or serialize all writes; document that read tools can still observe a mutation's transient
  state (the L4/L4-pass1 concern).

### R2 — the notify→loop bridge is an unbounded channel not drained under backpressure `[fix]`
`crates/watch/src/lib.rs:227` ✔ (unbounded confirmed) — sharper form of pass-1 M/L notes

The bridge is an **unbounded** channel; while the loop is parked on the bounded output send
(`out_tx.send(...).await`, `:544`), `drain()` runs only *after* the send completes.

- **Scenario:** `working_tree(true)` over a tree a build/indexer churns (10³–10⁴ events/s); the
  consumer keeps the `RepoWatcher` alive but stops calling `recv()` (e.g. only polls `stats()`) →
  the 64-slot output channel fills, the loop parks at `send`, and the unbounded `raw_rx` queue grows
  at the FS-event rate **indefinitely** — slow, unrecoverable memory growth in a daemon (no
  deadlock, but an unbounded leak, invisible to `stats()`).
- **Fix direction:** bound the raw bridge (drop-oldest with a `dropped` counter) or coalesce into a
  single "dirty" flag rather than queueing every event. **Landed (2026-07):** the bridge is now a
  capacity-1 `try_send` channel — a burst coalesces into one pending "re-check" (the loop re-queries
  the full snapshot anyway), so memory is bounded with no state loss.

### R3 — the watcher goes silently deaf if the OS watch dies (repo re-init / backend thread death) `[fix]`
`crates/watch/src/lib.rs:228` ◦

If the notify backend watch dies permanently — backend thread death, or the watched `.git`/`.jj`
dir is deleted-and-recreated (a re-clone / re-init) — the error callback fires at most once (one
final re-query), the watch is **never re-registered**, no error reaches the consumer, and `recv()`
yields nothing forever while `stats()` shows flat-but-healthy counters.

- **Scenario:** a prompt daemon watches a repo; the user re-clones it in place → the OS watch on the
  old inode dies → the watcher looks like a quiet repo for the rest of the process lifetime.
- **Fix direction:** surface a terminal `Error`/event on watch death (so the consumer can rebuild),
  and/or re-register on the parent dir.

### R-MEDIUM
- **R4** `crates/watch/src/lib.rs:248` — the **baseline** snapshot in `Builder::build` runs with
  **no** `requery_timeout`; a repo whose snapshot wedges (hung fsmonitor/network FS/jj lock) on a
  `Repo` built without `default_timeout` hangs the caller **at startup** — the exact failure the
  loop-side timeout was added for. `[fix: apply the deadline to the baseline too]` **Landed
  (2026-07):** `build()` now bounds the baseline capture with `requery_timeout`, returning a
  transient `Io` `TimedOut` instead of hanging.
- **R5** `crates/github/src/lib.rs:862` (`run_watch`) & `git/src/lib.rs:1465` clone/fetch — the
  **unbounded output buffer** (processkit default; no `output_buffer` call anywhere) means
  `gh run watch` (re-prints the full job table every ~3 s) accumulates tens of MB over a multi-hour
  run, all thrown away by `let _ =`; and clone/diff/`show_file` on a huge repo OOMs (this is pass-1
  **M30**, re-confirmed from the process angle). `[fix: wire with_max_bytes on the at-risk verbs]`
  **Partially landed (2026-07):** `run_watch`'s discarded buffer is now bounded (drop-oldest, last
  256 lines / 256 KiB) — a zero-downside fix (only the exit status is used). The content-verb /
  clone OOM is **deferred**: a hard-coded cap on `show_file`/`diff_text` would error on a legitimate
  large file, so it wants a *configurable* per-client output cap (a small API-design item), not a
  magic constant — tracked with M30.
- **R6** `crates/git/src/lib.rs:1465` (+ jj fetch) — retry × timeout amplification: `Error::Timeout`

## Second-pass synthesis

New systemic themes on top of the first pass's seven:

8. **Freeze-before-you're-ready (API).** Bare bools (A1), the input-rejection error model (A2),
   the resource-not-found gap (A3), transposable same-typed params (A5), and prose-only DTO honesty
   (A6) all become *permanent* the moment the traits hit 1.0. These are cheap now and breaking later
   — they gate the 1.0 tag, not a release.
9. **Release-automation can silently ship breakage (build).** The default `patch` bump can publish a
   too-narrow dependency floor (B1) or an API removal (B2), and 9/12 crates skip packaging validation
   (B3) — the safety net is a stale, report-only semver check. Promote semver-checks to a release
   gate and validate all 12 packages.
10. **Unbounded/steady-state growth in the long-lived processes (resource).** Unbounded output
    capture (R5), the undrained notify bridge (R2), and silent watcher death (R3) are the failure
    modes that only show up *after hours* in the MCP server / watcher — the two consumers that run
    longest. None panics; all degrade silently.

**Panic-discipline is a genuine strength** — worth preserving with a `clippy::unwrap_used`/
`expect_used` deny in library crates so it can't regress.
